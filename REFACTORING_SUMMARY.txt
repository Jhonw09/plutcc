ARCHITECTURE OVERVIEW
================================================================================

LAYERS:

┌─────────────────────────────────────────────────────────────────┐
│  UI COMPONENTS (DashboardPage, AuthForm, etc.)                 │
│  • No direct API calls                                          │
│  • Use hooks: useAuth(), useClass()                             │
│  • Display data + handle user interactions                      │
└───────────────────────────┬─────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────────┐
│  CUSTOM HOOKS (useAuth, useClass)                              │
│  • Manage loading/error state                                  │
│  • Simplify component logic                                    │
│  • Call services for data                                      │
└───────────────────────────┬─────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────────┐
│  CONTEXT + SERVICES (AuthContext, authService, classService)   │
│  • Contain all business logic                                  │
│  • Handle mock vs. real toggling                               │
│  • Manage persistence (localStorage)                           │
│  • Standardized error handling                                 │
└───────────────────────────┬─────────────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────────────┐
│  DATA SOURCES (Mock Data or Real API)                          │
│  • Mock: In-memory objects + simulated delay                   │
│  • Real: fetch() calls to backend HTTP endpoints               │
└─────────────────────────────────────────────────────────────────┘


HOW IT WORKS:

1. Component calls useClass() hook
2. Hook calls classService.joinClass()
3. Service checks: USE_MOCK flag?
   - If true: Return mock data (with simulated delay)
   - If false: Make real fetch() call
4. Service returns data or throws error
5. Hook catches error → stores in error state
6. Component receives result via hook state
7. Component renders

BENEFIT: Components don't know or care if data is mock or real!


================================================================================
KEY DIFFERENCES: BEFORE vs. AFTER
================================================================================

BEFORE (Direct fetch in components):
─────────────────────────────────────

// AuthContext.jsx
async function login({ email, senha }) {
  const res = await fetch(ENDPOINTS.login, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ email, senha }),
  })
  
  if (!res.ok) {
    if (res.status === 401) throw new Error('...')
    throw new Error('...')
  }
  
  const data = await res.json()
  // ... process
}

Problem:
  ✗ Can't switch to mock without refactoring
  ✗ No simulated network delay for testing
  ✗ API logic mixed with state management
  ✗ Hard to test


AFTER (Services with mock/real abstraction):
──────────────────────────────────────────────

// AuthContext.jsx
async function login({ email, senha }) {
  const data = await authService.login({ email, senha })
  // ... process
}

// authService.js
export const authService = {
  login: (credentials) =>
    USE_MOCK ? mockLogin(credentials) : realLogin(credentials),
}

Benefits:
  ✓ Single toggle to switch backends
  ✓ Mock automatically includes network delay
  ✓ API logic isolated in services
  ✓ Easy to test mock path
  ✓ Components unchanged
  ✓ Clear separation of concerns

HOW TO SWITCH FROM MOCK TO REAL BACKEND
================================================================================

QUICK START:

Step 1: Ensure backend is ready
  □ All API endpoints implemented
  □ Response format matches service expectations
  □ Backend runs on http://localhost:8080 (dev)
  □ Or set VITE_API_URL env var (production)

Step 2: Update src/api/services/config.js
  
  Line 6, change:
    export const USE_MOCK = true
  
  To:
    export const USE_MOCK = false

Step 3: Restart frontend
  $ npm run dev

Step 4: Test
  - Sign up with new account
  - Login
  - Join a class (if implemented)
  - Refresh page—session persists
  - Check Network tab in DevTools—see real fetch calls

Step 5: Debug issues
  - Browser console for JS errors
  - Network tab for HTTP errors
  - Backend logs for server errors
  - Check CORS if seeing network errors


BEFORE vs. AFTER BEHAVIOR:

Mock Mode (USE_MOCK = true):
  - Instant response (after 300–800ms delay)
  - Fixed error messages
  - No network errors possible
  - Data lost on refresh
  - Perfect for UI development

Real Mode (USE_MOCK = false):
  - Real network latency
  - Backend error messages
  - Network errors possible (CORS, timeout, etc.)
  - Data persisted in database
  - Production-ready

EXTENDING WITH NEW SERVICES
================================================================================

Pattern: Add exerciseService for managing exercises

1. Create src/api/services/exerciseService.js

2. Implement mock + real functions:

   async function mockGetExercises(userId, classId) {
     await simulateNetworkDelay()
     if (!userId) throw new Error('Sessão inválida...')
     return mockExercises.filter(e => e.classId === classId)
   }

   async function realGetExercises(userId, classId) {
     if (!userId) throw new Error('Sessão inválida...')
     const res = await fetch(`${ENDPOINTS.exercises}/${classId}`)
     if (!res.ok) throw new Error('Não foi possível...')
     return res.json()
   }

3. Add endpoint to config.js:

   export const ENDPOINTS = {
     // ... existing
     exercises: `${API_BASE}/exercicios`,
   }

4. Export service:

   export const exerciseService = {
     getExercises: (userId, classId) =>
       USE_MOCK ? mockGetExercises(userId, classId)
                : realGetExercises(userId, classId),
   }

5. Create hook (optional) src/hooks/useExercises.js:

   export function useExercises() {
     const { user } = useAuth()
     const [loading, setLoading] = useState(false)
     const [error, setError] = useState(null)
     
     const getExercises = useCallback(async (classId) => {
       setLoading(true)
       try {
         const result = await exerciseService.getExercises(user?.id, classId)
         return result
       } catch (err) {
         setError(err.message)
         throw err
       } finally {
         setLoading(false)
       }
     }, [user?.id])
     
     return { getExercises, loading, error }
   }

6. Use in component:

   function ExercisesPage() {
     const { getExercises, loading } = useExercises()
     
     useEffect(() => {
       getExercises(classId)
         .then(setExercises)
         .catch(err => console.error(err))
     }, [classId])
     
     return loading ? <Spinner /> : <ExercisesList />
   }

ERROR HANDLING STRATEGY
================================================================================

All errors are:
  1. User-friendly (Portuguese messages)
  2. Thrown from services
  3. Caught by hooks/context
  4. Stored in error state
  5. Displayed in UI (toasts, alerts)

Error Types:

Validation Errors (Service-side):
  throw new Error('Código de turma inválido.')
  throw new Error('Sessão inválida. Faça login novamente.')

HTTP Errors (Real backend):
  404 → throw new Error('Turma não encontrada.')
  409 → throw new Error('Você já está inscrito nesta turma.')
  500 → throw new Error('Erro no servidor. Tente novamente.')

Network Errors (Real backend):
  timeout → throw new Error('Erro de conexão. Tente novamente.')
  CORS → throw new Error('Erro ao comunicar com servidor.')

Component Usage:

throw → catch → error state → display in UI (no re-throw needed)

function JoinClassModal() {
  const { joinClass, error } = useClass()
  
  async function handleJoin(codigo) {
    try {
      await joinClass(codigo)
      // Success path
      onJoin(...)
    } catch (err) {
      // Error already stored in `error` state
      // Component displays: {error && <Alert>{error}</Alert>}
    }
  }
}
